🏗️ Visão Arquitetural

🤔 Pergunta Socrática

"Se cada serviço tem seu próprio banco de dados e nunca chama outro diretamente… como eles se comunicam?"

Através de eventos assíncronos. Quando um pedido é criado no Orders API, um evento é publicado no RabbitMQ. O Notification Worker consome esse evento sem que o Orders API saiba — ou se importe — com o que acontece depois.

01

Fundação e Estrutura

Clean Architecture · SharedKernel · EF Core · Docker Compose

🤔 Pergunta Socrática

"Como organizar um sistema para que trocar o ORM nunca quebre regras de negócio?"

A resposta é a Dependency Rule da Clean Architecture: camadas internas nunca conhecem as externas. O Domain não sabe que EF Core existe. Se amanhã trocarmos para Dapper ou um banco NoSQL, o Domain continua intacto.

Clean Architecture — as setas apontam para dentro

Conceitos-Chave

📦

📋

Serilog com propriedades tipadas. Pesquise por {OrderId}, não por regex.
🐳

⚠️ Anti-Padrão

Múltiplas versões do mesmo pacote entre projetos. Sem Central Package Management, o Identity usa EF Core 10.0.0 e o Orders usa 10.0.1 — bugs sutis em migrations que só aparecem em CI.

02

Domínio Rico e DDD

Aggregates · Value Objects · Domain Events · State Machine

🤔 Pergunta Socrática

"Por que um OrderItem nunca deveria existir sem uma Order?"

Porque Order é o Aggregate Root — a guardiã das invariantes. Se alguém adicionasse um OrderItem diretamente, poderia criar um item sem atualizar o total do pedido. O aggregate garante que toda mudança passa por regras de negócio.

Aggregate Order — tudo dentro da fronteira é atômico
// Rich Domain Model — comportamento encapsulado no aggregate
public sealed class Order : AggregateRoot
{
    private readonly List<OrderItem> _items = [];

    public void AddItem(Guid productId, Money unitPrice, int quantity)
    {
        if (_status != OrderStatus.Pending)
            throw new OrderDomainException("Can only add items to pending orders.");

        _items.Add(OrderItem.Create(Id, productId, unitPrice, quantity));
        AddDomainEvent(new OrderItemAddedDomainEvent(Id, productId, quantity));
    }
}
💼 Na Entrevista

"Qual a diferença entre Entity e Value Object?"

Entity tem identidade (Id) — duas entidades com mesmos dados mas IDs diferentes são objetos diferentes. Value Object é definido por seus atributos — dois Money(100, "BRL") são iguais, não importa onde estão. Consequência: Value Objects são imutáveis.

⚠️ Anti-Padrão — Anemic Domain Model

Order é só um saco de dados com getters/setters públicos. Toda lógica fica em OrderService. Resultado: invariantes violadas, OrderItem criado sem atualizar total, status muda sem validação. Se a entidade não protege a si mesma, ninguém protege.

03

CQRS com MediatR

Commands · Queries · Pipeline Behaviors · Dapper · Result Pattern

🤔 Pergunta Socrática

"A query que carrega um relatório precisa passar pela mesma validação complexa que cria um pedido?"

Não. E é exatamente isso que CQRS resolve: leitura e escrita têm necessidades diferentes. Escrita precisa de domínio rico, validação, transação. Leitura precisa de velocidade e projeções planas. No OrderFlow, escrita usa EF Core; leitura usa Dapper com SQL direto.

CQRS — pipelines diferentes por tipo de operação
// Pipeline Behavior — valida ANTES de abrir transação
public class ValidationBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
{
    public async Task<TResponse> Handle(
        TRequest request, CancellationToken ct,
        RequestHandlerDelegate<TResponse> next)
    {
        var result = await _validator.ValidateAsync(request, ct);

        if (!result.IsValid)
            return (TResponse)(object)Result.Failure(result.Errors[0].ErrorMessage);

        return await next();  // handler nunca recebe dados inválidos
    }
}
💡 Result Pattern > Exceptions

"Pedido não encontrado" não é excepcional — é esperado. Result<T> retorna Success(data) ou Failure(error) sem custo de stack unwinding. Exceções são para situações realmente excepcionais (banco fora, rede caiu).

🔬 Deep Dive — Sênior

Por que Dapper é mais rápido? EF Core faz change tracking: mantém snapshot de cada entidade carregada para detectar mudanças no SaveChanges(). Para leitura, isso é custo puro — você não vai salvar nada. Dapper executa SQL e materializa DTOs sem overhead. Em cenários de reporting, a diferença pode ser 5-10x.

04

Autenticação e Segurança

JWT Bearer · Refresh Token Rotation · Authorization Policies · Rate Limiting

🤔 Pergunta Socrática

"Se JWT é stateless e o servidor não armazena nada… como revogar o acesso de um usuário demitido?"

Você não revoga o JWT diretamente — ele expira em 15 minutos. O que você revoga é o Refresh Token. Sem refresh, o access token morre sozinho. E com Refresh Token Rotation, cada uso gera um novo par — se um token roubado for detectado (reuse), toda a família é revogada.

Refresh Token Rotation — cada uso gera novo par
// Refresh Token com detecção de roubo
public class RefreshToken
{
    public string Token { get; init; }
    public DateTime ExpiresAt { get; init; }
    public DateTime? RevokedAt { get; set; }
    public string? ReplacedByToken { get; set; }

    public bool IsActive => RevokedAt == null && ExpiresAt > DateTime.UtcNow;
}

// Se alguém tenta usar token já revogado → theft detected
// Revoga TODA a família de tokens do usuário
💼 Na Entrevista

"JWT vs Sessions — quando usar cada?"

JWT: stateless, escala horizontal sem shared state, ideal para APIs com múltiplos clients. Sessions: stateful, revogação instantânea, precisa de Redis/banco. No OrderFlow usamos JWT + Refresh para ter o melhor dos dois mundos.

05

Mensageria Assíncrona

RabbitMQ · MassTransit · Outbox Pattern · Consumers Idempotentes · DLQ

🤔 Pergunta Socrática

"O banco salvou o pedido com sucesso. Mas o RabbitMQ estava fora do ar naquele segundo. O evento se perdeu para sempre?"

Não, se você usar Outbox Pattern. Em vez de publicar o evento diretamente no broker, o evento é salvo na mesma transação que os dados do pedido. Um Background Service lê a tabela OutboxMessages e publica periodicamente. Se o broker falhar, a mensagem fica na tabela e tenta de novo.

Outbox Pattern — evento nunca se perde
// Consumer Idempotente — at-least-once exige deduplicação
public class OrderCreatedConsumer : IConsumer<OrderCreated>
{
    public async Task Consume(ConsumeContext<OrderCreated> ctx)
    {
        var msgId = ctx.MessageId.Value;

        // Já processou? Ignora (idempotência)
        if (await _db.IsProcessed(msgId))
            return;

        // Processa + marca como processado em transação
        await SendNotification(ctx.Message);
        await _db.MarkProcessed(msgId);
    }
}
💼 Na Entrevista

"Qual a diferença entre Domain Event e Integration Event?"

Domain Event é local (in-process, via MediatR) — OrderCreatedDomainEvent dispara handlers no mesmo serviço. Integration Event cruza fronteiras (inter-process, via RabbitMQ) — OrderCreated vai para outros serviços. Domain Events são síncronos; Integration Events são assíncronos.

06

Cache e Observabilidade

Redis Cache-Aside · Output Caching · OpenTelemetry · Serilog · Health Checks

🤔 Pergunta Socrática

"Como saber se seu cache está realmente ajudando — ou só consumindo memória?"

Medindo. O Cache Hit Ratio é a métrica fundamental:

📐 Fórmula

Hit Ratio = Hits / (Hits + Misses) × 100%

Alvo saudável: > 80% em endpoints de leitura frequente

// Decorator Pattern — Cache sem poluir o serviço original
public sealed class CachedProductService : IProductService
{
    private readonly IProductService _inner;
    private readonly IDistributedCache _cache;

    public async Task<Product> GetById(Guid id)
    {
        var key = $"product:{id}";
        var cached = await _cache.GetAsync(key);

        if (cached is not null)
            return JsonSerializer.Deserialize<Product>(cached)!; // HIT

        var product = await _inner.GetById(id); // MISS → busca no banco
        await _cache.SetAsync(key, ..., new { AbsoluteExpiration = 5.Minutes() });
        return product;
    }
}

Os 3 Pilares da Observabilidade

📊
Metrics (O quê)
Contadores, gauges, histogramas. "Quantos pedidos/min?" "Cache hit ratio?" Prometheus → Grafana.
🔗
Traces (Onde)
TraceId propaga entre serviços. "Orders API (200ms) → RabbitMQ (50ms) → Worker (100ms) = 350ms total." OpenTelemetry → Jaeger.
📝
Logs (Por quê)
Structured com Serilog. Log.Information("Order {OrderId} created", id) — pesquisável por propriedade no Seq.
⚠️ Anti-Padrão

Cache sem TTL. Dados obsoletos ficam no Redis indefinidamente. O produto muda de preço no banco mas o cache serve o preço antigo por horas. Sempre defina TTL + invalidação por eventos.

07

API Gateway e Docker

YARP Reverse Proxy · Docker Multi-stage · Compose · Testcontainers

🤔 Pergunta Socrática

"Seu cliente precisa conhecer 5 URLs diferentes para usar seu sistema. Se você mudar a porta de um serviço, todos os clients quebram. Como resolver?"

API Gateway. O YARP é um ponto único de entrada: http://gateway:8080/api/orders/* roteia para o Orders API internamente. O client conhece uma URL. Se um serviço mudar de porta, atualiza a config do gateway — clients nem percebem.

🔬 Deep Dive — Testcontainers

Em vez de mockar SQL Server e RabbitMQ nos testes de integração, Testcontainers sobe containers reais descartáveis. O teste roda migrations no banco real, publica eventos no RabbitMQ real, e verifica comportamento end-to-end. Se funciona no teste, funciona em produção — porque é o mesmo banco.

08

CI/CD e Deploy na Azure

GitHub Actions · Azure Container Apps · Bicep IaC · Environments

🤔 Pergunta Socrática

"Você confia o suficiente no seu código para que, a cada push, ele vá automaticamente para produção?"

Esse é o espectro Continuous Delivery vs. Continuous Deployment. Delivery: deploy automático até staging, com aprovação manual para produção. Deployment: automático até produção. No OrderFlow usamos Delivery — confiança com human gate.

Pipeline CI/CD — do push ao deploy com approval gate
# .github/workflows/cd.yml
name: CD
on:
  push:
    branches: [main]

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - run: docker build -t ghcr.io/orderflow/api:${{ github.sha }} .
      - run: docker push ghcr.io/orderflow/api:${{ github.sha }}
      - run: az containerapp update --image ghcr.io/orderflow/api:${{ github.sha }}

  deploy-production:
    needs: deploy-staging
    environment: production  # ← requer aprovação manual
    steps:
      - run: az deployment group create --template-file infra/main.bicep
💼 Na Entrevista

"Container Apps vs AKS — quando usar cada?"

Container Apps: serverless, define imagem + scaling rules e pronto. Sem gerenciar nodes. Ideal até ~50 serviços. AKS (Kubernetes): controle total — service mesh, GPU, custom operators. Complexidade enorme. OrderFlow usa Container Apps porque não precisa de customização profunda.
