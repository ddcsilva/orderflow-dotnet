  2. Se dois projetos na mesma solution usam versões diferentes do mesmo NuGet — o que pode dar errado em produção?

2.1 Por Que Clean Architecture para o Catalog?

🧠 Analogia — O Edifício Corporativo

Imagine um prédio comercial bem planejado. O subsolo (Domain) tem a fundação e os cofres — ninguém de fora toca lá. O térreo (Application) é a recepção: coordena quem entra e sai, mas não guarda nada de valor. Os andares de escritório (Infrastructure) são onde o trabalho pesado acontece — banco de dados, emails, serviços externos. E a fachada (Api) é o que o público vê — bonita, mas se você demolir e reconstruir, o prédio continua de pé.

Cada andar tem uma responsabilidade clara e se comunica por portas e elevadores (interfaces), não por buracos na parede.

Clean Architecture — Regra de Dependência

O Catalog é um serviço CRUD. Para um CRUD puro, Clean Architecture pode parecer overengineering — e de fato seria em um projeto de vida curta. Mas estamos usando aqui por três motivos:

  1. Didático — Praticamos a separação de camadas em um contexto simples antes de aplicar no Orders (que é complexo de verdade)
  2. Consistência — Todos os serviços seguem a mesma estrutura, facilitando onboarding
  3. Preparação para escala — CRUDs simples frequentemente ganham regras de negócio com o tempo. Começar organizado evita a refatoração dolorosa
🏛️ Alternativa — Vertical Slice Architecture

Para CRUDs que realmente permanecerão simples, Vertical Slice Architecture (pasta por feature) é mais prática — menos cerimônia e acoplamento naturalmente baixo entre features. Veja a seção Aprofundamento Sênior.

2.3 Por Que Central Package Management?

🧠 Analogia — O Cardápio Único do Restaurante

Imagine que você tem 8 filiais de um restaurante. Cada filial compra seus próprios ingredientes, com marcas e versões diferentes. O prato "estrogonofe" fica diferente em cada lugar. O Directory.Packages.props é como um caderno de compras central: todas as filiais usam a mesma marca de creme de leite, a mesma versão do molho. Se precisar trocar fornecedor, muda num lugar só.

Benefícios concretos:

  • Uma única fonte de verdade — PR de atualização de pacotes toca um arquivo só
  • Atualizações atômicas — dotnet outdated + uma linha alterada = todos os projetos atualizados
  • Conflitos impossíveis — não existe cenário onde projeto A usa uma versão e projeto B outra acidentalmente
  • Segurança — Vulnerabilidade em um pacote? Atualiza uma linha, rebuilda tudo

2.5 Por Que Serilog (e não ILogger nativo)?

🧠 Analogia — Ficha de Atendimento vs Bilhete Rabiscado

Imagine um hospital. Um médico pode rabiscar "paciente com dor" num post-it (log não-estruturado). Ou pode preencher uma ficha padronizada: nome, data, sintomas, setor, prioridade (log estruturado). Quando você precisa filtrar "todos os pacientes com dor no setor B nas últimas 2 horas", o post-it é inútil — a ficha permite consultas instantâneas. Structured logging é a ficha. String concatenada é o post-it.

Serilog é o provider mais popular no ecossistema .NET por:

  • Structured logging — Cada propriedade vira um campo pesquisável
  • Sinks — Console, File, Seq, Elasticsearch, Application Insights, Grafana Loki...
  • Enrichers — Adiciona automaticamente CorrelationId, MachineName, ThreadId
  • Templates — Formato de saída customizável por ambiente

Código limpo: Seus services continuam usando ILogger<T> (a abstração do .NET). Serilog é configurado apenas no Program.cs. Se amanhã quiser trocar por OpenTelemetry Logging ou NLog, muda um arquivo.

🔬 Docker Compose — O que é, para que serve e o que cada configuração significa

O que é Docker Compose? É uma ferramenta que define e orquestra múltiplos containers com um único arquivo YAML. Em vez de rodar docker run com 15 flags para cada serviço, você declara tudo no docker-compose.yml e sobe tudo com docker compose up -d.

Para que serve no OrderFlow? Em vez de instalar SQL Server e Seq na sua máquina (o que polui o sistema operacional e cria conflitos de versão), cada serviço roda isolado em seu container. Quando não precisar mais, docker compose down remove tudo sem deixar rastro.

SQL Server:

image: mcr.microsoft.com/mssql/server:2022-latestImagem oficial do SQL Server 2022 da Microsoft. mcr = Microsoft Container Registry
container_name: orderflow-sqlserverNome fixo do container — facilita referência em logs e comandos
ports: "1433:1433"Mapeia porta 1433 do host para 1433 do container. localhost:1433 no seu código se conecta ao SQL Server dentro do container
ACCEPT_EULA: "Y"Aceita a licença do SQL Server (obrigatório)
MSSQL_SA_PASSWORD: "${SA_PASSWORD:-...}"Senha do admin. Lê do arquivo .env ou usa o valor padrão após :-
MSSQL_PID: "Developer"Edição Developer — gratuita com todas as features do Enterprise
volumes: sqlserver-data:/var/lib/mssql/dataNamed volume — dados persistem mesmo se o container for recriado
healthcheck: test: sqlcmd ... "SELECT 1"Tenta executar SELECT 1 a cada 10s. Se falhar 5x, container é unhealthy. start_period: 30s dá tempo para inicializar
restart: unless-stoppedReinicia automaticamente se crashar — exceto se você fez docker compose stop

Seq (Log Server):

image: datalust/seq:latestImagem oficial do Seq — UI web para visualizar logs estruturados
ports: "5341:80"Porta 80 interna mapeada para 5341. Acesse http://localhost:5341
ACCEPT_EULA: "Y"Aceita a licença (Seq é gratuito para desenvolvimento single-user)
volumes: seq-data:/dataPersiste os logs mesmo se recriar o container

Named Volumes são gerenciados pelo Docker e ficam em \\wsl$\docker-desktop-data\... (Windows). São mais seguros que bind mounts porque o Docker controla permissões e lifecycle.

Arquivo .env: O Docker Compose lê automaticamente o .env no mesmo diretório. Isso evita hardcoding de senhas no YAML — o .env fica no .gitignore.

Comandos essenciais:

4.1 SharedKernel — Classes Base

🧠 Analogia — O DNA da Aplicação

O SharedKernel é como o DNA compartilhado entre irmãos. Cada serviço (Catalog, Orders, Identity) é uma "pessoa" diferente, mas todos carregam os mesmos genes fundamentais: o que é uma Entity, o que é um DomainEvent, como fazer Equals. O SharedKernel garante que as regras fundamentais são escritas uma vez e herdadas por todos.

⚠️ Armadilha Comum

Cuidado para não transformar o SharedKernel em um "lixão compartilhado". Ele deve conter apenas conceitos universais. A regra de ouro: "Se eu remover um serviço inteiro, o SharedKernel continua fazendo sentido?"

📂 Por que os arquivos estão todos na raiz, sem subpastas?

O SharedKernel tem apenas 5 arquivos (Entity.cs, AuditableEntity.cs, IDomainEvent.cs, IRepository.cs, IUnitOfWork.cs), todos no mesmo namespace OrderFlow.SharedKernel. Criar pastas como Entities/, Interfaces/, Events/ para 1-2 arquivos cada seria over-engineering — adiciona navegação desnecessária sem benefício real. A regra prática: crie pastas quando tiver 5+ arquivos do mesmo tipo. Quando o SharedKernel crescer (ValueObject base, Result pattern, novos eventos), reorganize. Até lá, a raiz plana é a solução mais simples que funciona (KISS).

namespace OrderFlow.SharedKernel;

public abstract class Entity
{
    public Guid Id { get; protected init; }

    private readonly List<IDomainEvent> _domainEvents = [];

    public IReadOnlyCollection<IDomainEvent> DomainEvents => _domainEvents.AsReadOnly();

    protected void AddDomainEvent(IDomainEvent domainEvent)
    {
        _domainEvents.Add(domainEvent);
    }

    public void ClearDomainEvents()
    {
        _domainEvents.Clear();
    }

    public override bool Equals(object? obj)
    {
        if (obj is not Entity other) return false;
        if (ReferenceEquals(this, other)) return true;
        if (Id == Guid.Empty || other.Id == Guid.Empty) return false;
        return Id == other.Id;
    }

    public override int GetHashCode() => Id.GetHashCode();

    public static bool operator ==(Entity? left, Entity? right) => Equals(left, right);
    public static bool operator !=(Entity? left, Entity? right) => !Equals(left, right);
}
🔬 Dissecando o Código — Entity

abstract impede instanciação direta — você nunca cria new Entity(), apenas entidades concretas. É o Template Method Pattern.

protected init no Id — init permite atribuir valor apenas durante inicialização, protected restringe a subclasses. Imutabilidade seletiva.

IReadOnlyCollection<T> externamente, List<T> internamente — é o princípio do menor privilégio aplicado a coleções.

Identity Equality (DDD) — entidades são iguais pelo Id, não pelos campos. Guard clause para Guid.Empty impede que entidades não-persistidas sejam consideradas iguais.

🔬 Interface Segregation Principle (ISP)

Uma única propriedade intencional — o contrato mínimo que todo evento de domínio deve cumprir: "quando aconteceu?". Metadata adicional será adicionada por enrichers, não pela interface.

Princípio YAGNI — "You Aren't Gonna Need It": YAGNI diz: não implemente algo até que você realmente precise. É tentador antecipar funcionalidades futuras ("e se um dia precisarmos de X?"), mas isso gera código que ninguém usa e aumenta a complexidade. O custo de manter código desnecessário é sempre maior que o custo de adicioná-lo quando for preciso. Resolva o problema de hoje com a solução mais simples possível.

Por que não herdar de MediatR.INotification agora? O SharedKernel é base de todos os serviços. Se adicionarmos MediatR aqui, forçamos uma dependência transitiva em projetos que não precisam dele (ex: Catalog). Na Fase 3 (CQRS), quando o Orders precisar, atualizaremos a interface. Isso segue o princípio YAGNI e mantém o SharedKernel leve.

using OrderFlow.SharedKernel;

namespace OrderFlow.Catalog.Domain.Entities;

public sealed class Product : AuditableEntity
{
    public string Name { get; private set; } = string.Empty;
    public string? Description { get; private set; }
    public string Sku { get; private set; } = string.Empty;
    public decimal Price { get; private set; }
    public int StockQuantity { get; private set; }
    public bool IsActive { get; private set; } = true;

    public Guid CategoryId { get; private set; }
    public Category? Category { get; private set; }

    private Product() { } // EF Core

    public static Product Create(
        string name, string sku, decimal price,
        int stockQuantity, Guid categoryId, string? description = null)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(name);
        ArgumentException.ThrowIfNullOrWhiteSpace(sku);
        if (price < 0) throw new ArgumentException("O preço não pode ser negativo.");
        if (stockQuantity < 0) throw new ArgumentException("A quantidade em estoque não pode ser negativa.");

        return new Product
        {
            Id = Guid.NewGuid(), Name = name.Trim(),
            Sku = sku.Trim().ToUpperInvariant(),
            Price = price, StockQuantity = stockQuantity,
            CategoryId = categoryId, Description = description?.Trim(),
            CreatedAt = DateTime.UtcNow
        };
    }

    public void Update(string name, string? desc, decimal price, int stock)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(name);
        if (price < 0) throw new ArgumentException("O preço não pode ser negativo.");
        if (stock < 0) throw new ArgumentException("A quantidade em estoque não pode ser negativa.");
        Name = name.Trim(); Description = desc?.Trim();
        Price = price; StockQuantity = stock;
        SetUpdated();
    }

    public void ChangeCategory(Guid newCategoryId) { /* validates + sets */ SetUpdated(); }
    public void Deactivate() { IsActive = false; SetUpdated(); }
    public void Activate() { IsActive = true; SetUpdated(); }

    public bool HasSufficientStock(int qty) => StockQuantity >= qty;

    public void DecreaseStock(int quantity)
    {
        if (!HasSufficientStock(quantity))
            throw new InvalidOperationException(
                $"Estoque insuficiente. Disponível: {StockQuantity}, Solicitado: {quantity}");
        StockQuantity -= quantity;
        SetUpdated();
    }

    public void IncreaseStock(int quantity)
    {
        if (quantity <= 0) throw new ArgumentException("A quantidade deve ser positiva.");
        StockQuantity += quantity;
        SetUpdated();
    }
}
🏛️ Lógica de Estoque no Domínio

DecreaseStock e IncreaseStock vivem na entidade, não no service. Estoque é uma regra de invariante — "nunca pode ficar negativo". Colocando na entidade, é impossível violar a regra independente de quem chama.

Command-Query Separation (CQS): HasSufficientStock apenas consulta (query), DecreaseStock apenas muda estado (command).

public sealed record PagedResult<T>(
    IReadOnlyList<T> Items, int TotalCount, int Page, int PageSize)
{
    public int TotalPages => (int)Math.Ceiling((double)TotalCount / PageSize);
    public bool HasPreviousPage => Page > 1;
    public bool HasNextPage => Page < TotalPages;
}
🏛️ Records como DTOs — Imutabilidade por Design

Separação de concerns: ProductDto = output (com Id, CategoryName computado). CreateProductRequest = input (sem Id — servidor gera). UpdateProductRequest = input parcial (sem Sku — imutável após criação). PagedResult<T> = record genérico com computed properties.

💡 Em palavras simples — O que o PagedResult faz e para que serve?

Imagine que sua loja tem 10.000 produtos. Quando o usuário abre a página de produtos, você não quer devolver todos de uma vez — seria lento, gastaria memória e a tela ficaria impossível de navegar. A solução é paginação: devolver os produtos em "páginas", tipo 20 por vez.

O PagedResult<T> é justamente o "envelope" que carrega uma página de resultados. Ele guarda:

  • Items — a lista de itens daquela página (ex: os 20 produtos da página 3)
  • TotalCount — o total de itens que existem no banco (ex: 10.000)
  • Page — em qual página você está (ex: 3)
  • PageSize — quantos itens cabem por página (ex: 20)

E calcula automaticamente:

  • TotalPages — quantas páginas existem no total (10.000 ÷ 20 = 500 páginas)
  • HasPreviousPage — se existe página anterior (página 3 → sim)
  • HasNextPage — se existe próxima página (página 3 de 500 → sim)

O <T> (genérico) significa que funciona com qualquer tipo: PagedResult<ProductDto> para produtos, PagedResult<CategoryDto> para categorias — o mesmo código serve para todos. É como um envelope padrão dos Correios: funciona para qualquer carta, não precisa de um envelope diferente para cada destinatário.

src/Services/Catalog/OrderFlow.Catalog.Application/Services/ProductService.cs
public sealed partial class ProductService(
    IProductRepository productRepository,
    ICategoryRepository categoryRepository,
    IUnitOfWork unitOfWork,
    IValidator<CreateProductRequest> createValidator,
    IValidator<UpdateProductRequest> updateValidator,
    ILogger<ProductService> logger) : IProductService
{
    public async Task<ProductDto> CreateAsync(
        CreateProductRequest request, CancellationToken ct = default)
    {
        // 1. Validate input (fail fast)
        var validation = await createValidator.ValidateAsync(request, ct);
        if (!validation.IsValid)
            throw new ValidationException(validation.Errors);

        // 2. Verify category exists
        var category = await categoryRepository.GetByIdAsync(request.CategoryId, ct)
            ?? throw new KeyNotFoundException($"Categoria com ID '{request.CategoryId}' não foi encontrada.");

        // 3. Check SKU uniqueness
        if (await productRepository.SkuExistsAsync(request.Sku, ct))
            throw new InvalidOperationException($"Já existe um produto com o SKU '{request.Sku}'.");

        // 4. Create domain entity
        var product = Product.Create(
            request.Name, request.Sku, request.Price,
            request.StockQuantity, request.CategoryId, request.Description);

        // 5. Persist (atomic via Unit of Work)
        await productRepository.AddAsync(product, ct);
        await unitOfWork.SaveChangesAsync(ct);

        // 6. Structured logging (source-generated)
        LogProductCreated(logger, product.Id, product.Name);

        return MapToDto(product);
    }

    public async Task DeleteAsync(Guid id, CancellationToken ct = default)
    {
        var product = await productRepository.GetByIdAsync(id, ct)
            ?? throw new KeyNotFoundException($"Produto com ID '{id}' não foi encontrado.");

        product.Deactivate();           // Soft delete!
        productRepository.Update(product);
        await unitOfWork.SaveChangesAsync(ct);
    }

    [LoggerMessage(Level = LogLevel.Information, Message = "Product created: {ProductId} - {ProductName}")]
    private static partial void LogProductCreated(ILogger logger, Guid productId, string productName);
    // ... GetByIdAsync, GetBySkuAsync, SearchAsync, UpdateAsync ...
}
🏛️ Application Service — O Orquestrador

ProductService não contém lógica de negócio — ele orquestra: valida input → busca entidades → chama métodos do domínio → persiste → retorna DTO.

Primary constructor (C# 12) + partial — parâmetros viram campos implícitos. O partial é necessário para os métodos [LoggerMessage] source-generated. Validate-first — falha rápido antes de qualquer I/O. Null-coalescing throw (??) verifica nulidade em uma linha.

Soft delete — DeleteAsync chama Deactivate(), não Remove(). Combinado com HasQueryFilter do EF Core, o produto "desaparece" das queries mas permanece no banco.

🔬 LoggerMessage Source Generation

LogProductCreated(logger, product.Id, product.Name) — gerado em compile-time pelo atributo [LoggerMessage]. Evita boxing de parâmetros e alocações em runtime. Com TreatWarningsAsErrors=true, os warnings CA1848/CA1873 bloqueiam o build se usar logger.LogInformation() diretamente. Os templates como {ProductId} capturam valores como propriedades estruturadas no Serilog.

public sealed class ProductRepository(CatalogDbContext context) : IProductRepository
{
    public async Task<Product?> GetByIdAsync(Guid id, CancellationToken ct)
    {
        return await context.Products
            .Include(p => p.Category)     // JOIN no SQL
            .FirstOrDefaultAsync(p => p.Id == id, ct);
    }
    // GetByIdAsync SEM AsNoTracking — intencional para write queries!

    public async Task<(IReadOnlyList<Product>, int)> SearchAsync(
        string? searchTerm, Guid? categoryId,
        decimal? minPrice, decimal? maxPrice,
        int page, int pageSize, CancellationToken ct)
    {
        var query = context.Products
            .Include(p => p.Category)
            .AsNoTracking()      // Read-only: economiza memória
            .AsQueryable();       // Deferred execution

        // Filtros condicionais — cada if adiciona WHERE
        if (!string.IsNullOrWhiteSpace(searchTerm))
            query = query.Where(p =>
                p.Name.Contains(searchTerm) ||
                p.Sku.Contains(searchTerm));
        if (categoryId.HasValue)
            query = query.Where(p => p.CategoryId == categoryId.Value);
        if (minPrice.HasValue)
            query = query.Where(p => p.Price >= minPrice.Value);
        if (maxPrice.HasValue)
            query = query.Where(p => p.Price <= maxPrice.Value);

        var totalCount = await query.CountAsync(ct);
        var items = await query
            .OrderBy(p => p.Name)
            .Skip((page - 1) * pageSize)  // Paginação offset-based
            .Take(pageSize)
            .ToListAsync(ct);             // Aqui a query vai ao banco!

        return (items, totalCount);
    }

    public async Task<bool> SkuExistsAsync(string sku, CancellationToken ct)
    {
        var normalizedSku = sku.Trim().ToUpperInvariant();
        return await context.Products
            .IgnoreQueryFilters()  // Inclui inativos!
            .AnyAsync(p => p.Sku == normalizedSku, ct);
    }
}

    public async Task<IActionResult> Update(
        Guid id, [FromBody] UpdateProductRequest request, CancellationToken ct)
        => Ok(await productService.UpdateAsync(id, request, ct));

    [HttpDelete("{id:guid}")]
    public async Task<IActionResult> Delete(Guid id, CancellationToken ct)

🏛️ Controller Magro, Service Gordo

Controller não tem lógica — apenas roteia HTTP para Service. [Route("api/v1/[controller]")] — versionamento na URL. {id:guid} — route constraint rejeita IDs inválidos antes do código.

CreatedAtAction retorna HTTP 201 com header Location — padrão REST. CancellationToken é injetado automaticamente pelo ASP.NET Core quando o client desconecta.

public sealed partial class GlobalExceptionHandler(
    ILogger<GlobalExceptionHandler> logger) : IMiddleware
{
    private static readonly JsonSerializerOptions JsonOptions = new(JsonSerializerDefaults.Web);

    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        try { await next(context); }
        catch (Exception ex) { await HandleExceptionAsync(context, ex); }
    }

    private async Task HandleExceptionAsync(HttpContext ctx, Exception ex)
    {
        var (statusCode, errorResponse) = ex switch
        {
            ValidationException v =>
                (HttpStatusCode.BadRequest,
                 new ErrorResponse("Erro de Validação",
                     v.Errors.Select(e => e.ErrorMessage).ToArray())),

            KeyNotFoundException n =>
                (HttpStatusCode.NotFound,
                 new ErrorResponse("Não Encontrado", [n.Message])),

            InvalidOperationException i =>
                (HttpStatusCode.Conflict,
                 new ErrorResponse("Violação de Regra de Negócio", [i.Message])),

            _ => (HttpStatusCode.InternalServerError,
                  new ErrorResponse("Erro Interno do Servidor",
                      ["Ocorreu um erro inesperado."]))
        };

        // Erro 500 = LogError; Erro 4xx = LogWarning
        if (statusCode == HttpStatusCode.InternalServerError)
            LogUnhandledException(logger, ex, ex.Message);
        else
            LogHandledException(logger, ex, (int)statusCode);

        ctx.Response.StatusCode = (int)statusCode;
        ctx.Response.ContentType = "application/json";
        await ctx.Response.WriteAsJsonAsync(errorResponse, JsonOptions);
    }

    [LoggerMessage(Level = LogLevel.Error, Message = "Unhandled: {Message}")]
    private static partial void LogUnhandledException(ILogger logger, Exception ex, string message);

    [LoggerMessage(Level = LogLevel.Warning, Message = "Handled: {StatusCode}")]
    private static partial void LogHandledException(ILogger logger, Exception ex, int statusCode);
}

public sealed record ErrorResponse(string Title, string[] Errors);
🔒 Segurança — Information Disclosure

Para erro 500, a mensagem é genérica: "Ocorreu um erro inesperado." — nunca exponha stack traces ao client (OWASP). O log completo vai para Serilog/Seq. Erros 4xx usam LogWarning (esperados), não LogError — evita alertas falsos. A classe é partial para suportar os métodos [LoggerMessage] gerados em compile-time. JsonSerializerOptions é cachedo em static readonly para evitar alocações repetidas (CA1869).

07. Perguntas de Entrevista — Fase 1

"O que é Clean Architecture e como difere de camadas tradicionais?"
Na arquitetura tradicional, a dependência flui de cima para baixo: Presentation → Business → Data. O problema: Business depende de Data. Na Clean Architecture, a dependência aponta para dentro: Api → Infrastructure → Application → Domain. O Domain é puro — zero dependências externas. Você pode trocar SQL Server por PostgreSQL sem alterar lógica de negócio.
"Por que Central Package Management e não versões no .csproj?"
Com 15+ projetos, gerenciar versões em cada .csproj vira pesadelo. O Directory.Packages.props centraliza todas as versões. Atualizar uma dependência = uma linha. É o equivalente ao Maven BOM ou gradle.properties no mundo Java.
"Diferença entre Controllers e Minimal APIs?"
Controllers: classes com rotas declarativas, filtros, model binding robusto, versionamento fácil. Minimal APIs: delegates lambda, menos cerimônia, mais performance, ideal para microserviços focados. Mostre que domina ambos.
"Explique EF Core Code First com Migrations."
Define entidades C# + Fluent API. O EF Core compara o modelo com o banco e gera uma migration (Up() e Down()). dotnet ef database update aplica. Schema versionado no Git, reproduzível em qualquer ambiente. Nunca edite migration já aplicada em produção — crie uma nova.
"Por que SQL Server e não PostgreSQL para .NET?"
SQL Server é o banco mais pedido nas vagas .NET no Brasil (bancos, fintechs, consultorias). EF Core suporta ambos igualmente. A escolha depende de custo (PostgreSQL é gratuito) e features específicas.

A4. Database per Service — Disciplina Crítica

🚫 Nunca compartilhar DB entre microserviços

Sintomas de violação: Migration de um serviço quebra outro, performance degrada, schema vira monólito disfarçado.

Solução: Para "preciso de dado de outro serviço" → eventos (Fase 05) ou API (gRPC, Fase 13). Nunca SQL cross-service.

Perguntas Sênior de Entrevista

"Quando abandonaria Clean Architecture por Vertical Slice?"
Quando o sistema é CRUD-heavy, time é grande com features paralelas, e o overhead de 4 camadas + repositórios não paga ROI.
"Como evolui schema sem downtime?"
Padrão Expand → Migrate → Contract: (1) adicionar nova coluna nullable, (2) deploy app que escreve em ambas, (3) migrar dados em background, (4) deploy app que lê só da nova, (5) remover coluna antiga. Nunca faz RENAME ou DROP em colunas usadas.
